# ADR-009: 署名付き音声 URL の失効に備え再生直前にエピソードを再取得する

**ステータス:** 採用済み
**決定日:** 2026-06-14
**対象読者:** iOS 開発者
**関連:** [iOS設計書 §8（音声再生）](../design/ios-design.md)

## 背景

`GET /podcasts` / `GET /podcasts/{id}` が返す `audio_url` は GCS の**署名付き URL で有効期限 1 時間**（バックエンドの既定 `expiration_seconds=3600`）。一覧取得から時間が経った後に再生しようとすると、URL が失効し再生に失敗しうる。Web は再生開始フロー（`useStartPodcast`）で再生直前にエピソードを再取得することでこれを回避している。

## 検討した選択肢

| 選択肢 | メリット | デメリット |
|---|---|---|
| 一覧取得時の `audio_url` をそのまま再生に使う | 実装が単純 | 1 時間経過後に失効し再生失敗。長時間アプリを開いたままの実機で頻発 |
| 失効を検知してリトライ（エラー後に再取得） | 通常時の通信が減る | 失敗 → 再取得 → 再生で初回再生が遅延。エラー判定が URL 失効か他要因か曖昧 |
| **再生直前に必ず `GET /podcasts/{id}` で再取得** | 常に新鮮な URL で確実に再生。Web と同方針 | 再生のたびに 1 リクエスト増（軽量で許容範囲） |

## 決定

エピソード再生開始時に `GET /podcasts/{id}` で当該エピソードを再取得し、得られた新しい `audio_url` を `AVPlayerItem` に渡す。一覧で保持していた URL は再生には使わない。

## 理由

署名付き URL の失効は時間依存で予測困難なため、再生直前の再取得が最も確実。追加コストは 1 リクエストのみで、Web の確立した方式とも一致するため。

## 結果（影響）

- 再生フローは「再取得 → ローカル保存位置へ seek → 再生」の順で実装する。
- 長時間再生中に URL が失効するケース（1 時間超の連続再生）は稀だが、Phase 2 でローカルキャッシュ（ダウンロード）導入時に併せて対処する。
- バックエンドが将来 `audio_url` の有効期限を延長/恒久化した場合、本 ADR を見直す。
