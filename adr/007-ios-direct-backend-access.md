# ADR-007: iOS はバックエンド API へ直接接続する（BFF プロキシを介さない）

**ステータス:** 採用済み
**決定日:** 2026-06-14
**対象読者:** iOS 開発者・バックエンド開発者
**関連:** [ADR-001](001-web-bff-proxy.md)（Web の BFF プロキシ）

## 背景

Web は CORS 回避とユーザー実行時設定の両立のため BFF プロキシ（`/api/backend/*`）を経由する（[ADR-001](001-web-bff-proxy.md)）。iOS ネイティブアプリのネットワークアクセスにはブラウザのオリジン制約（CORS / プリフライト）が適用されないため、同じ制約は発生しない。一方でバックエンドは `X-API-Key` 認証 + デプロイ単位の `USER_ID` による単一ユーザー前提である。

## 検討した選択肢

| 選択肢 | メリット | デメリット |
|---|---|---|
| Web と同様に BFF プロキシ層を別途用意 | Web と経路を揃えられる | iOS には不要な中継。運用コスト増・レイテンシ増。ネイティブに CORS は無関係 |
| **バックエンドへ直接接続（X-API-Key 直送）** | 構成が最小・追加インフラ不要・低レイテンシ | バックエンド URL/キーを端末が保持（[ADR-008](008-ios-local-state-persistence.md) で対処） |

## 決定

iOS は `AppState` が保持するベース URL に対し `URLSession` で**直接**リクエストし、`X-API-Key` ヘッダーを付与する。BFF プロキシは利用しない。

## 理由

CORS はネイティブに無関係で、プロキシを挟む技術的必然性がない。直接接続が最も単純で、追加インフラ・レイテンシを生まないため。

## 結果（影響）

- iOS のベース URL は実機から到達可能なホストを指す（Web のローカル Docker 名 `http://api:8080` のような事情は対象外）。
- バックエンドの単一ユーザー前提（`USER_ID`）は不変。マルチユーザー化は本決定のスコープ外。
- エラー正規化はクライアント側 `APIError` の責務（Web の `ApiError` と同方針）。
- 将来バックエンドに CORS が追加されても iOS の経路は変わらない。
