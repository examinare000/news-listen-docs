# ADR-040: iOS Editorial デザインシステムの採用

**ステータス:** 採用済み
**決定日:** 2026-06-30
**対象読者:** iOS 開発者・デザイン担当
**関連:** `agent-rules/15-frontend-design.md`（フロントエンドデザイン規約）・[ios-design-system.md](../design/ios-design-system.md)（正本）・[ADR-034](034-ios-accessibility-voiceover-dynamic-type.md)（アクセシビリティ）・[issue #110](../../issues/110)

## 背景

iOS アプリの UI は白基調・システムフォント（SF）・標準 `List`/`Form` の素な組み合わせで、視覚的個性が乏しく、
コア価値（厳選ニュースを耳で深掘りする読み物体験）に見合う品位・没入感を欠いていた。プロジェクトのフロントエンド
デザイン規約 `15-frontend-design.md` は、汎用フォント・紋切り型配色・クッキーカッターな汎用テンプレートを禁じ、
美的方向性の選択肢として「エディトリアル」を挙げている。Web には視覚正本 `app-ui.html` があるが、iOS には同等の
デザイン基盤も視覚仕様ドキュメントも存在しなかった。

## 検討した選択肢

| 選択肢 | メリット | デメリット |
|---|---|---|
| **Editorial デザインシステム（採用）** | 読み物体験に合う品位・没入感。トークン化で直値散在を解消し一貫性を担保。セリフ見出しで個性を出しつつ Apple 純正書体で依存ゼロ | トークン整備と全画面適用の初期コスト |
| システム標準のまま（現状維持） | 追加コストなし・常に OS 標準に追従 | 無味乾燥・没個性。規約 15 の禁止事項に抵触 |
| サードパーティ UI キット導入 | 既製コンポーネントで高速 | 依存増・アプリの個性が出ない・規約の「汎用感回避」に反する |

## 決定

iOS に **Editorial（雑誌風）デザインシステム**を採用し、`DesignSystem/` 配下にトークンを集約する。

- **配色**: 温かみのある紙（paper）＋墨（ink）を支配色に、朱（accent）を唯一のアクセントとする。状態色（danger/success/star）も
  トークン化し直値（`.red` 等）を排除。すべてライト/ダーク2値の適応色。
- **書体**: 見出しは Apple 純正セリフ（New York, `design: .serif`）、本文・メタは SF。テキストスタイル基準で Dynamic Type に追従。
- **適用**: Feed・再生・Podcast・設定・アカウント・ログイン・オンボーディング・ユーザー管理の各画面へ一貫適用。
- **アクセシビリティ非後退**: Dynamic Type・VoiceOver を維持。`onAccent` を適応色にしてダークモードのコントラストを確保。

詳細なトークン仕様・適用方針は [ios-design-system.md](../design/ios-design-system.md) を正本とする。

## 結果

- 色・書体・余白の直値が View から排除され、トークン参照に統一された（保守性・一貫性の向上）。
- ライト/ダーク両対応。明色アクセント上の前景コントラスト不足（旧 `.white` 固定）を `onAccent` 適応色で是正。
- 実装は `DSColor` / `DSFont` / `DSLayout` / `DSAppearance` / `Components/`。issue #110 の受け入れ条件を充足。
