# ADR-001: Web はバックエンド API へ BFF プロキシ経由でアクセスする

**ステータス:** 採用済み
**決定日:** 2026-06-10（2026-06-12 文書化）
**対象読者:** Web フロントエンド開発者

## 背景

バックエンド（FastAPI / Cloud Run）には CORS ミドルウェアが存在せず、ブラウザ JS からのクロスオリジン fetch はプリフライトで失敗する。また API のベース URL はユーザーが実行時に設定する仕様（localStorage 保存）のため、ビルド時に転送先を固定できない。

## 検討した選択肢

| 選択肢 | メリット | デメリット |
|---|---|---|
| バックエンドに CORSMiddleware を追加 | プロキシ不要で構成が薄い | backend の変更・再デプロイが必要になり、Web 実装がバックエンド作業をブロック要因に持つ（1 ブランチ = 1 目的に反する） |
| Next.js rewrites | 実装が最小 | 転送先 URL がビルド時固定。実行時のユーザー設定と矛盾 |
| **キャッチオール Route Handler による BFF プロキシ** | CORS が発生しない。転送先をリクエストヘッダーで実行時指定できる | プロキシ層の実装・テストが必要 |

## 決定

`web/app/api/backend/[...path]/route.ts` にキャッチオール Route Handler を実装し、ブラウザは常に同一オリジンの `/api/backend/*` を呼ぶ。クライアントは `X-API-Key` と `X-Backend-Base-Url` ヘッダーを付与し、プロキシがサーバー側でバックエンドへ転送する。

## 理由

CORS 問題を web 単独の変更で解決でき、転送先の実行時指定とも両立する唯一の案のため。

## 結果（影響）

- ステータスコード・レスポンスボディは加工せず素通し（エラー正規化はクライアント `ApiError` の責務）
- SSRF 緩和のため `X-Backend-Base-Url` は `http(s)://` スキーム以外を 400 で拒否
- ローカル Docker では SetupModal のバックエンド URL に `http://api:8080`（コンテナ間名前解決）を入力する
- Phase 2 でバックエンドに CORS を追加すればプロキシは除去可能（ADR で廃止を記録すること）
