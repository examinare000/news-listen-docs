# ADR-017: パスワード強度バリデーションとログイン時セッションローテーション

**ステータス:** 採用済み
**決定日:** 2026-06-24
**対象読者:** バックエンド / Web / iOS 開発者
**関連:** [ADR-013](013-session-auth-and-user-management.md)（セッション認証の詳細）・[ADR-014](014-login-rate-limiting.md)（レートリミットと対で総当たり耐性を高める）・[ADR-015](015-audit-logging.md)（監査ログ）・[backend 設計書（認証）](../design/backend-design.md#6-認証ユーザー管理adr-013)・`agent-rules/12-security-guidelines.md`・backend PR #19（issue #10）

## 背景

[ADR-013](013-session-auth-and-user-management.md) で bcrypt ハッシュ化とセッション認証を、[ADR-014](014-login-rate-limiting.md) でログイン試行レートリミットを導入した。これらは「保存時の保護」と「総当たりの抑止」を担うが、**弱いパスワードを作らせない前段防御**と**固定セッション（session fixation）リスクの低減**は未対応だった。

- パスワードは**最小長 8 のみ**（`api/schemas.py`）で、複雑性・既知流出パスワードのチェックが無い。利用者が `password123` のような弱い資格情報を設定できてしまう。
- ログイン成功時に**セッショントークンの再発行（ローテーション）が無い**。ログイン前に提示されたセッション ID が認証後も生き続けうるため、固定セッション攻撃の余地が残る。

決定すべき論点は (A) 強度ポリシーの内容と既知流出パスワード拒否の方式、(B) 強度検証の適用範囲、(C) セッションローテーションの方式、の 3 点。

## 検討した選択肢

### (A) 既知流出パスワードの拒否方式

| 選択肢 | メリット | デメリット |
|---|---|---|
| **オフライン同梱ブロックリスト（採用）** | ネットワーク非依存・決定的（テスト容易）・低レイテンシ | 網羅性は同梱リスト次第。巨大リストはイメージを肥大化 |
| HIBP k-匿名 API | 広範な流出パスワードを照合可能 | 外部ネットワーク依存・レイテンシ・テストの非決定性（要モック）。1〜5 人規模には過剰 |
| 複雑性チェックのみ | 実装最小 | 既知流出語（複雑性は満たすが流出済み）を拒否できない |

### (B) 強度検証の適用範囲

| 選択肢 | メリット | デメリット |
|---|---|---|
| ログインを含む全パスワード入力に適用 | 一貫 | **既存ユーザーがログイン不能**になる（過去に作られた弱い PW を弾く）。重大な回帰 |
| **新規・変更パスワードのみに適用（採用）** | 弱い PW の新規作成を防ぎつつ既存ログインを壊さない | スキーマごとに適用箇所を選ぶ必要 |

### (C) セッションローテーション方式

| 選択肢 | メリット | デメリット |
|---|---|---|
| 同時セッション数の上限導入 | 漏洩時の被害局限 | 本 issue の主眼ではない。スコープ拡大 |
| **ログイン成功時に旧トークンを失効し新規発行（採用）** | 固定セッション対策を最小実装で達成。既存の `delete_session` ポートを再利用 | 同時セッション数の制御は別途 |

## 決定

- **(A) オフライン同梱ブロックリストを採用**する。`shared/data/common_passwords.txt` を同梱し、`shared/password_policy.py` がモジュールロード時に 1 回だけ読み込み `frozenset` 化する（小文字正規化）。ファイル不在時は fail-open（他ルールは機能）。
- **強度ポリシー**（純粋関数 `validate_password_strength(password, *, username=None)`、定数で集約）:
  - 最小長 **12 文字**（従来 8 から引き上げ）。
  - 文字クラス **4 種（小文字・大文字・数字・記号）のうち 3 種以上**。
  - ブロックリスト不該当（小文字照合）。
  - **ユーザー名の非包含**（前後空白を除去して照合。等価、または username 長 4 以上のとき部分一致を拒否。短い username は等価のみ）。
  - bcrypt の 72 バイト上限（`_PASSWORD_MAX`）は維持。判定は**文字数**で行う。
  - 違反時は `ValueError` を送出し、**メッセージに平文パスワードを含めない**。
- **(B) 強度検証は新規・変更パスワードにのみ適用**する。`UserCreateRequest.password`・`PasswordChangeRequest.new_password`・`UserUpdateRequest.new_password` に Pydantic バリデータで適用。**`LoginRequest.password` は対象外**（`min_length=1` を維持）とし、既存ユーザーのログインを壊さない。`UserCreateRequest` は username を同一モデルで持つため username 非包含検査を行う。
- **(C) ログイン成功時にセッションローテーション**する。`api/routers/auth.py` の `login` で、**認証成功確定後・新トークン発行前**に、リクエストが既存セッショントークンを提示していれば `db.delete_session(hash_token(old_token))` で旧セッションを失効させる。新トークン発行・`create_session`・Cookie 発行は ADR-013 のまま。冪等（旧トークン無し・不在セッションでも安全）。
- **強度しきい値は環境変数化しない**（定数固定）。運用者が緩い値を注入してセキュリティを後退させられないようにする。

## 理由

- 1〜5 人規模では、オフラインの最小ブロックリスト＋複雑性で「明らかに弱い／既知流出のパスワード」を実用十分に弾ける。HIBP API のネットワーク依存・非決定性を負う便益は小さい。
- ログインに強度検証を掛けると既存ユーザーが締め出される。新規・変更時のみに適用することで、防御を前進させつつ後方互換を保つ。
- ローテーションを**認証成功後**に置くことが要。仮に資格情報検証より前に旧セッション削除を行うと、攻撃者が誤った資格情報と被害者のトークンを送るだけで被害者のセッションを失効させられてしまう。成功後・新発行前に置くことで、固定化を防ぎつつこの悪用を排除する。
- しきい値の env 化は「緩める」方向の事故を招く。セキュリティポリシーは定数で固定する方が安全かつ単純。

## 結果（影響）

- **新規モジュール/データ**: `shared/password_policy.py`（`validate_password_strength`）、`shared/data/common_passwords.txt`（同梱ブロックリスト）。`Dockerfile.api` は `shared/` を同梱済みのため配布に追加対応は不要。
- **スキーマ**: `api/schemas.py` で `_PASSWORD_MIN` を 8→1 に下げ（実下限はポリシーが強制し、Field は空文字のみ拒否）、`UserCreateRequest`/`PasswordChangeRequest`/`UserUpdateRequest` に強度バリデータを追加。`UserUpdateRequest.new_password` は `None`（変更なし）を許容。
- **API**: `api/routers/auth.py` の `login` にローテーション処理を挿入。`_extract_session_token` / `hash_token` は既存利用。
- **エラー応答の機微情報秘匿**: FastAPI 既定の検証エラーは送信値（`input`）を本文に載せるため、`api/main.py` に `RequestValidationError` ハンドラを追加し、機微フィールド（`password`/`new_password`/`current_password`）の `input`/`ctx` を伏せる。弱いパスワードを送っても **422 応答本文に平文が反映されない**。
- **テスト**: `tests/test_password_policy.py`（境界長・文字クラス・ブロックリスト・username 検査・unicode・多バイト・メッセージ非漏洩）、`tests/test_schemas.py`（各スキーマの 422）、`tests/test_api_auth.py`（ローテーション・失敗時非ローテーション・ログイン免除）、`tests/test_api_admin_users.py`（422 で平文非エコー）。強度引き上げに伴い、既存テストの作成/変更パスワードを強い値へ更新。
- **クライアント（Web/iOS）**: パスワード設定 UI で強度要件（12 文字・3 文字クラス）をユーザーに提示するのが望ましい（UX。サーバ側 422 を待たずに前段で案内）。本 ADR の必須範囲外。
- **残課題**: **同時セッション数の上限**は本 ADR の対象外（follow-up）。セルフサービスのパスワードリセット（メール）は引き続き範囲外。MFA は将来課題。
